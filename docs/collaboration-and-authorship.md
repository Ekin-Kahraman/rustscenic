# Collaboration and Authorship

This document records the default expectations for collaborations around rustscenic. It is not a legal agreement. It exists so roles are explicit before the project becomes larger.

## Project ownership

rustscenic is an open-source software project created and maintained by Ekin Kahraman.

The code is MIT licensed, so others may use, copy, modify, and redistribute it under the licence terms. The licence does not remove the need for accurate attribution, citation, or authorship when the software is used in scientific work.

## Software credit

Work that depends on rustscenic should cite the repository or the exact release used. Derivative software should preserve the MIT licence and copyright notice.

For papers, talks, posters, grants, or lab reports, the software contribution should be described accurately. Creating and maintaining the software is distinct from applying it to a dataset.

## Validation and data contributions

Labs or collaborators who provide datasets, biological interpretation, benchmark runs, or failure reports should be credited for those contributions. These contributions can be substantial and may justify authorship on papers that depend on them.

Validation credit should not overwrite software credit. Software development, benchmark design, data generation, biological interpretation, and manuscript writing should each be named separately.

## Manuscripts

Before starting a paper, collaborators should agree in writing on:

- expected first author, co-first author, senior author, and corresponding author roles
- who owns each part of the work
- which datasets can be used publicly
- which affiliation and compute resources are being used
- what work is still required before submission
- how changes in contribution level affect authorship

For a software paper primarily about rustscenic itself, the default expectation is that the creator and lead developer is the lead software author unless another contributor materially changes the intellectual centre of the work.

## Practical rule

If a collaborator's role changes from feedback or validation into major writing, analysis, direction, or supervision, authorship and project roles should be revisited immediately rather than after a manuscript is drafted.
